# バックログ層 診断レポート

**問い**：「バックログがあまり機能してない感じがするのは、物語の途中でアプデしてるからか？」
**対象**：第1章のバックログ層（プロダクト/スプリントバックログ＋カンバン）
**結論を一言で**：途中アプデは*従因*。主因は **バックログ層がゲームの結末（エンディング）にほぼ影響しない**こと。物語と「会話していない」設計が効きすぎている。

---

## 1. TL;DR（3行）

1. ゲームは**2つの独立レイヤー**で動く：①ルーレット＝物語（3メーターを動かす）、②バックログ＝カンバン（顧客価値バーを動かす）。`backlog.ts:2-3` に「daily ルーレットとは独立した別レイヤー」と明記。
2. **エンディングは3メーター（trust/insight/culture）だけで決まる**（`chapter-01.ts:130-158`）。顧客価値（北極星）は*表示*のみで結末を分岐させない。
3. バックログ層から結末に効く唯一の経路は、スプリントレビュー時の **culture ±1 ナッジ**（`backlog.ts:239`）だけ。全3スプリントで最大 ±3。→ プレイヤーは「盤面を触っても話が変わらない」と*正しく*直感している。これが"機能してない感"の正体。

---

## 2. 二層モデル

```
┌─ ルーレット層（物語）──────────────┐      ┌─ バックログ層（カンバン）──────────┐
│ daily で spin → イベント → 選択      │      │ planning: 並べ替え提案＋forecast     │
│ → 3メーター(trust/insight/culture)   │      │ daily: 着手(AI生成)→レビュー→Done    │
│   が動く                            │      │ → delivery / coverage を積む          │
│ progression.spinCore / chooseCore   │      │ engine/backlog.ts                     │
└──────────────┬──────────────────┘      └──────────────┬──────────────────┘
               │                                          │
        （結末を決める）                          （表示バーを動かす）
               │                                          │
               ▼                                          ▼
        ENDINGS.match(m)  ◀── 3メーターのみ        customerValue 北極星バー
        chapter-01.ts:130                          （結末には未接続）
```

**重み（`customerValueBreakdown` / `progression.ts:752`）**：means(3メーター)=60、delivery(Done数/DELIVERY_TARGET=6)=20、coverage=20、debt penalty=減算。
→ バックログ層は北極星の最大40点を占める“ように見える”。だが北極星は結末を分岐しない。さらに coverage は「会心実行」でルーレット層からも積める（`progression.ts:458-459`）ので、**バックログ専有なのは delivery(最大20) のみ**。

## 3. 結線インベントリ（両層をつなぐ橋＝これしかない）

| # | 橋 | 実装箇所 | 強さ | 備考 |
|---|----|---------|------|------|
| B1 | `discoversPbi`：ヒアリングの好実行で発見PBIをバックログに追加 | `progression.ts:525` / events-sprint1 の **3箇所のみ** | 弱 | Sprint2/3 には無い。一方向（物語→盤面）。 |

## 4. 切断点（なぜ"機能してない"感が出るか）— 優先度順

- **🔴 C1：盤面操作が結末を動かさない。** ENDINGS は3メーターのみ（`chapter-01.ts:130-158`）、北極星は表示のみ。バックログから結末に効くのは culture ±最大3 だけ。→ プレイヤーの「触っても話が変わらない」直感は正しい。**これが主因。**
- **🔴 C2：完全に任意・非ブロッキング。** 物語は「▶ 進める」のルーレット軸だけで完走できる。バックログは Board の任意ボタン（`Board.tsx:296-322`）で、計画も着手もスキップ可能。→ "開いても開かなくても進む側パネル"。
- **🟡 C3：物語が盤面を参照しない（B-逆方向ゼロ）。** 何を forecast したか・何を Done したかが、その後のイベントに反映されない。→ 盤面が“真空で回っている”読後感。
- **🟡 C4：橋が Sprint1 に偏在。** discoversPbi は3つとも Sprint1（`events-sprint1.ts:85,555,892`）。Sprint2/3 では物語→盤面の発見が消える。→ 後半ほど盤面が空気化。

---

## 5. ご指摘の「途中でアプデ」＝従因（読後感を悪くしている部分）

盤面が*プレイヤー操作と独立に作り替わる*3つの仕掛け。仕様としては正しいが、噛み合いが弱いため“勝手に変わる”ノイズに見える：

- **D1：発見PBIの途中注入**（`progression.ts:525` `revealPbi`）— 計画した盤面に項目が増える。
- **D2：スプリント末の全クリア**（`backlog.ts:254-265`）— forecast/inProgress/reviewProgress をリセット。
- **D3：キャリーオーバー戻し**（`backlog.ts:227, 264`）— 未完が翌スプリントのバックログに戻る。

→ C1/C2 で“効かない”と感じている状態に D1〜D3 の churn が重なると、「自分の操作が残らない・勝手に変わる」体感を増幅する。
