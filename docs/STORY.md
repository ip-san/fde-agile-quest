## 0. ねらい（なぜこの物語か）

プログラマーを **FDE（Forward Deployed Engineer＝客先の現場に深く入り込み、開発も段取りも自分でやり切る
エンジニア）** へリスキリングする学習ゲーム。FDE心得100箇条・スクラム・物流ドメインを「講義でなく、
判断の追体験で“いつの間にか”身につける」。判断＝主、用語・物語＝従。

## 1. 世界観・中心事件（“フェニックス”）

- **主人公（プレイヤー）**: SaaSベンダー「**ルーメン**」で在庫最適化システム「**StockPilot**」の裏側を5年
  書いてきた技術者。客先に会ったことがない。社の方針転換で**第一号のFDE**に指名される。
- **危機**: 売上（ARR）が半年横ばい、解約が増え、AIで似たソフトが誰でも作れる時代。「**SaaS is dead**」。
  社長**鷹野**は「ソフトを売って終わり」を捨て、客先に人を送り込み成果を共創するFDEへの転換を宣言。
- **潜入先**: フラッグシップ顧客の中堅物流会社「**カルゴ物流**」。StockPilotを鳴り物入りで導入したのに、
  倉庫の現場では誰も使っていない。情シスの**結城係長**は経営に「在庫を減らす」と約束済みで引けない。
  解約は目前。失えばルーメンの転換は頓挫し人員整理——失敗の許されない案件。
- **問い**: 沈黙する基幹システムを、現場の言葉でもう一度動かせるか。

### 企業の階層と力学（プレッシャーの源泉）

カルゴ物流は、グループ企業の末端にいる:

```
ジェネリックロボティクス（産業用ロボット大手・グループ総本山）
   └ ジェネリックロジスティクス（物流グループ・中間持株/事業会社）
        └ カルゴ物流（中堅物流・倉庫＝本作の舞台）
```

- **ジェネリックロボティクス**: 産業用ロボットの大手。**フィジカルAI**（現実世界で動く身体性を持ったAI＝
  ロボット×AI）へ手を伸ばそうとしているが**うまくいっていない**。**政府系ファンドの支援**を受けて延命・再起を図る。
- **ジェネリックロジスティクス**: その物流部門。グループのフィジカルAIを“現場で実証する場”を欲している。
- **「ジェネリック（汎用／コモディティ）」**という社名は、本作のテーマ（「汎用SaaSは誰でも作れる＝SaaS is
  dead」「汎用ロボットも価値が薄れる」）と響き合う。コモディティ化に追われる旧来型ハードの巨人、の含み。

**この階層が物語にもたらす力学（プロット・フック）**:
- **上からの自動化圧力**: グループはカルゴ物流の倉庫を「フィジカルAIのショーケース」にしたい。
  → 主軸の **トップダウン経路**（現場の意向を飛ばして自動化を急ぐ）の動機づけになる。郷田専務／本社の圧力の背景。
- **政府系ファンドの目**: 成果を出さないと支援が切れる＝失敗が許されない理由がグループ規模で存在する。
  レビュー（郷田専務／経営報告）の緊張感の源泉。
- **コスト vs 現場**: 赤城部長の「3PL移管・コスト削減」も、グループ再建の台所事情と地続き。
- **ルーメンとの接点（含み）**: ルーメンのFDE転換が、巨大グループの“現場でAIを動かす”課題と噛み合えば、
  将来の章（フィジカルAI×現場）への伏線になり得る。

> 注意（矛盾チェック）: 現状の第1章イベント本文は**カルゴ物流の現場**に閉じており、この階層は**背景設定**。
> 章/イベントに出すときは「フィジカルAI＝現実で動くAI（ロボット）」を平易に補足し、政府系ファンドは
> “上からの成果プレッシャー”として抽象的に扱う（実在の特定ファンドを断定しない）。

> 設定の地続き性（矛盾チェックの要）: 「現場が使わない不用品＝StockPilot＝主人公が作ったもの」。
> プロローグ／s1-daily-silentui／s2-retro 等はこの一本の線でつながっている。

## 3. メーター設計の原則（レビューで最重要のチェック軸）

3つのゲージ（各0..10、開始 **trust5 / insight4 / culture4**）＝3つの相手との関係:
- **顧客の信頼(trust)** … 結城さん／郷田専務など顧客
- **現場理解(insight)** … 田淵さん・現場
- **巻き込み(culture)** … チーム・橋本さん・文化

**不変ルール（全イベントが従う。矛盾＝原則違反としてチェック）**:
1. **信頼が下がるのは「顧客に摩擦が生じた」時だけ**（要望後回し／納期割れ／粗いものを見せる／耳の痛い事実／
   押し返し／案件を放る／面子を潰す）。**理由を必ず結果文に書く**。
2. チーム/プロセス系の“良い選択”は信頼を下げない。トレードオフは**機会コスト**＝もう一方の“速く見せる”
   選択が trust+1（進捗が見える）を持ち、正解はそれを**取り逃す**形（結果文に「（…の信頼+は取り逃す）」）。
3. **warn 選択肢**は「即時の負効果」か「将来の手戻り(setsFlag)」のどちらかの**下振れ**を必ず持つ。
4. **★0ルール**: どれか1ゲージが0で即バッドエンド（あるある失敗エピローグ）。

## 4. 構造（スクラムのリズム）

- キャンペーン = **Sprint 1 → 2 → 3**。各スプリント = `プランニング → デイリー×5 → レビュー → レトロ`。
- **デイリーだけルーレット**でセグメント抽選（現場/顧客/チーム/トラブル/チャンス）→プールから1問。
  **プランニング/レビュー/レトロは単発**（配列順で最初の出せるイベントを必ず出す＝決定論的）。
- **選択後ミニゲーム**: 開発（タイミング型／手順並べ替えパズル）か、ヒアリング（深掘り質問を選ぶ）を
  セグメントで出し分け。出来(great/good/poor)が**選んだ判断の主正メーターを +1/±0/−1** する（倍率）。
  負効果は不変＝0ルール/トレードオフを壊さない。スキップ＝good。

| Sprint | タイトル | ゴール |
|---|---|---|
| 1 | 現場を知る | 沈黙の理由を突き止める |
| 2 | 仮説を形にする | 誤出荷を減らす最小版を出す |
| 3 | 文化を残す | 成果を本番と人に根付かせる |

## 5. 主軸: ノウハウを隠す現場の自動化（信頼 ↔ トップダウンで分岐）

現場（田淵・山田・橋本）は、自動化＝自分の価値や仕事が奪われる恐れから、勘・暗黙知・**例外処理**を
わざと言葉にしない／隠す。FDEはそれを自動化する任務。**進め方で展開が変わる（毎回必ず分岐）**。

**分岐の仕組み**: 背骨（単発セレモニー）に `requiresFlag` バリアントを既定より前に置き、`proceedCore` が
フラグ一致のバリアントを優先表示＝決定論的。フラグは `s2-retro` で排他的にどちらか一方が立つ。

```
[Sprint1] s1-daily-hideknowhow  「見て覚えるもん」と隠す入口（主軸の提示）
        ↓
[Sprint2 レトロ] s2-retro  ★分岐点（毎回必ず選ぶ）
   ├─ a トップダウンで乗り切る  → flag: topDown   （trust+1 経営の覚え/速い・culture-1, warn）
   └─ b 信頼を築く・一緒に標準化 → flag: genbaTrust（insight+1, culture+1, 機会コストで trust+1 取り逃す）
        ↓
[Sprint3 レビュー]  経路で別イベント
   ├─ topDown   → s3-review-topdown  隠れた例外が漏れ本番デモが崩れる大障害（取り繕う=悪化 / 頭を下げる=挽回の入口）
   └─ genbaTrust → s3-review-trust    現場の勘が組み込まれ例外まで回る（手柄独占=warn / 現場の功績として語る）
        ↓
[Sprint3 レトロ]  経路で結末
   ├─ topDown   → s3-retro-topdown  挽回: 非を認め建て直す(culture+1 部分挽回) / 現場のせいにする(culture-2, 属人化エンドへ)
   └─ genbaTrust → s3-retro-trust    信頼の築き方をチームの財産に＝文化が根を張る
```

- **トップダウンの結末＝手痛い失敗→挽回の余地**（0ルール維持。未挽回は低culture／fail-cultureに寄る）。

## 6. その他の分岐（フラグチェーン）

| フラグ | 立てる所 | 回収する所 | 体験 |
|---|---|---|---|
| `wrongKpi` | s2-plan-kpi a（機能数をKPIに） | s3-daily-rework（要wrongKpi） | 誤ったWhy→後で全部やり直す手戻り |

## 7. 部門間対立（池井戸潤テイスト・社内政治）

「正論を通すと政治的に損をする」ジレンマ。媚びる側＝trust+1(短期評価)の機会コスト、筋を通す側＝
culture/insight+（時に顧客側の面子を潰して trust−、理由は結果文に明記）。代表:
- s2: 責任の押し付け合い(blamewar) / 「人を減らして数字を作れ」(costcut) / 本社一律通達 vs 現場(hqorder)
- s3: 手柄の横取り(credit) / 下請けへの値下げ要求(subcon) / 「地方に飛ばすぞ」派閥の脅し(faction)

## 8. エンディング（メーター駆動・単一最適解なし）

**通常クリア（Sprint3終了時、上から最初に一致）**:
| id | 条件 | タイトル |
|---|---|---|
| disliked | trust ≤ 2 | 現場に嫌われたFDE |
| orderTaker | insight ≤ 3 | 言われた通り作る人 |
| hero | culture ≤ 2 かつ trust ≥ 4 | ヒーロー止まり（属人化・文化を残せず） |
| trueFde | trust ≥ 7 かつ insight ≥ 6 かつ culture ≥ 6 | 真のFDE |
| decent | 上記以外 | 及第点のFDE |

**0ルール失敗（道中いずれかが0で即）**:
- trust0 → 「もう、来なくていいです」（出入り禁止）
- insight0 → 「結局あれ、何だったんですかね」（使われないシステム）
- culture0 → 「あの人がいないと、何も分からない」（属人化して文化が枯れる）

## 9. レビュー（矛盾チェック）の観点

物語/設定の一貫性は次を確認（多くは自動テストでも担保）:
- **キャラ呼称の一貫**（情シス→結城／ベテラン→田淵 等。表記ゆれが無いか）。
- **設定の地続き**（StockPilot＝主人公の製品＝使われない不用品、という線が崩れていないか）。
- **メーター原則**（§3）への適合：信頼−は顧客摩擦時のみで理由つきか／warn に下振れがあるか／
  両選択肢が trust+1 等の“支配”になっていないか（機会コスト型か）。
- **フラグ配線**（§5,6）：立てる→回収の前後関係、背骨バリアントの配列順、排他性。
- **エンディング整合**（§8）：閾値と物語のトーンが噛み合うか。
- **用語**：本文の `{{用語}}` が glossary に存在するか（テストで強制）。
- **心得**：各イベントに心得タグ、100箇条カバレッジ維持（テストで強制）。
