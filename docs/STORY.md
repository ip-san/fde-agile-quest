## 0. ねらい（なぜこの物語か）

プログラマーを **FDE（Forward Deployed Engineer＝客先の現場に深く入り込み、開発も段取りも自分でやり切る
エンジニア）** へリスキリングする学習ゲーム。FDE心得100箇条・スクラム・物流ドメインを「講義でなく、
判断の追体験で“いつの間にか”身につける」。判断＝主、用語・物語＝従。

## 1. 世界観・中心事件（“フェニックス”）

- **主人公（プレイヤー）**: SaaSベンダー「**ルーメン**」で在庫最適化システム「**StockPilot**」の裏側を5年
  書いてきた技術者。客先に会ったことがない。社の方針転換で**第一号のFDE**に指名される。
- **危機**: 売上（ARR）が半年横ばい、解約が増え、AIで似たソフトが誰でも作れる時代。「**SaaS is dead**」。
  社長**鷹野**は「ソフトを売って終わり」を捨て、客先に人を送り込み成果を共創するFDEへの転換を宣言。
- **潜入先**: フラッグシップ顧客の中堅物流会社「**カルゴ物流**」。StockPilotを鳴り物入りで導入したのに、
  倉庫の現場では誰も使っていない。情シスの**結城係長**は経営に「在庫を減らす」と約束済みで引けない。
  解約は目前。失えばルーメンの転換は頓挫し人員整理——失敗の許されない案件。
- **問い**: 沈黙する基幹システムを、現場の言葉でもう一度動かせるか。

### 企業の階層と力学（プレッシャーの源泉）

**カルゴ物流は、もともと独立した中堅物流会社**だった。ジェネリックのグループ生え抜きではない。
総本山 **ジェネリック電機**が、**フィジカルAIの“実証の場”として買収した**——というのが表向きの理由。
だが現実は皮肉だ。**カルゴ物流は基幹システムを入れても誰も使わない“沈黙する現場”**——実務は紙と
属人運用のままで、まともなデータ化すらできていない。フィジカルAIなど夢のまた夢で、実証は一向に進まない。そこでグループは、数字を取り繕うため、買収したカルゴ物流を**架空利益計上（粉飾）
に使う**。これが舞台の裏にある暗い力学だ。

```
ジェネリック電機（総合電機・産業機器の大手・グループ総本山）
   └ ジェネリックロジスティクス（物流グループ・中間持株/事業会社）
        └ カルゴ物流（中堅物流・倉庫＝本作の舞台。★元は独立企業 → フィジカルAI実証用に買収）
```

- **ジェネリック電機**: 総合電機・産業機器の大手。**フィジカルAI**（現実世界で動く身体性を持ったAI＝
  ロボット×AI）へ手を伸ばすが**うまくいっていない**。**政府系ファンドの支援**で延命・再起を図る。
  「フィジカルAIで物流を変える」と打ち上げ、その実証用にカルゴ物流を買った。
- **壮大な夢 vs アナログな現実**: 買ったはいいが、カルゴ物流は**システムを入れても現場が使わず、実態として
  データ化されていない**。フィジカルAIの前に、まず基本のIT化・自動化を“根付かせる”（＝FDEの仕事）ことが要る。
  夢と現実の落差が、上層を焦らせる。
- **架空の利益計上（粉飾＝架空循環取引）**: 実証が進まず数字が立たないので、ジェネリック電機の**フィジカル
  AI機材を“導入”名目でカルゴ物流に売り、グループ各社を書類の上だけで巡らせる**（同じ機材が一周するたびに
  売上が立つ＝**架空循環取引**）。だがカルゴ物流の現場は実数がデータ化されておらず、帳簿を突き合わせる者も
  いない——その機材は実在しない（**幽霊設備**）。
  ＝**「本物の成果」より「見栄えの数字」が欲しい**本音。手口の型は実在の循環取引型粉飾に着想（§詳細は不正暴露アーク）。
- **ルーメンとの契約**: 主人公の会社 **ルーメンは、カルゴ物流と契約**している（StockPilot導入の延長＋FDE派遣）。
  FDEはこの契約のもとで現場に入り、まず基本のIT化・自動化から手をつける。
- **「ジェネリック（汎用／コモディティ）」**という社名は、テーマ（「汎用SaaSは誰でも作れる＝SaaS is dead」
  「汎用ロボットも価値が薄れる」）と響き合う。コモディティ化に追われる旧来型ハードの巨人、の含み。

**この階層が物語にもたらす力学（プロット・フック）**:
- **数字の粉飾 vs 本物の価値（最重要の縦糸）**: 上層が欲しいのは**見栄えのする数字**。FDEがやるのは
  **誤出荷率を本当に下げる**こと。この対立は既存テーマ「数字を盛る vs 正直に」（s3-review 等）と直結する。
  → 「成果を盛って見せろ」という圧力の出どころ＝粉飾を支えたいグループ。正直＝grpの台所を脅かす。
- **上からの自動化圧力**: グループはカルゴ物流の倉庫を「フィジカルAIのショーケース」にもしたい。
  → 主軸の **トップダウン経路**（現場の意向を飛ばして自動化を急ぐ）の動機づけ。郷田専務／本社の圧力の背景。
- **政府系ファンドの目**: 成果（と粉飾された数字）が崩れると支援が切れる＝失敗が許されない理由がグループ規模で存在。
  レビュー（郷田専務／経営報告）の緊張感の源泉。買収された“外様”の現場は、その駒として使われている。
- **被買収企業ならではの距離感**: カルゴ物流の現場（田淵・橋本ら）にとって、グループも本社も“よそ者”。
  ノウハウを隠す動機（§5）に、「自動化への恐れ」だけでなく「買収した連中に明け渡したくない」も重ねられる。

> 注意（矛盾チェック）: 現状の第1章イベント本文は**カルゴ物流の現場**に閉じており、この階層・粉飾は**背景設定**。
> 章/イベントに出すときは「フィジカルAI＝現実で動くAI（ロボット）」「架空の利益計上＝数字の粉飾」を平易に補足し、
> 政府系ファンドは“上からの成果プレッシャー”として抽象的に扱う（実在の特定ファンド・企業・事件を断定しない）。

> 設定の地続き性（矛盾チェックの要）: 「現場が使わない不用品＝StockPilot＝主人公が作ったもの」。
> プロローグ／s1-daily-silentui／s2-retro 等はこの一本の線でつながっている。

## 3. メーター設計の原則（レビューで最重要のチェック軸）

3つのゲージ（各0..10、開始 **trust5 / insight4 / culture4**）＝3つの相手との関係:
- **顧客の信頼(trust)** … 結城さん／郷田専務など顧客
- **現場理解(insight)** … 田淵さん・現場
- **巻き込み(culture)** … チーム・橋本さん・文化

**不変ルール（全イベントが従う。矛盾＝原則違反としてチェック）**:
1. **信頼が下がるのは「顧客に摩擦が生じた」時だけ**（要望後回し／納期割れ／粗いものを見せる／耳の痛い事実／
   押し返し／案件を放る／面子を潰す）。**理由を必ず結果文に書く**。
2. チーム/プロセス系の“良い選択”は信頼を下げない。トレードオフは**機会コスト**＝もう一方の“速く見せる”
   選択が trust+1（進捗が見える）を持ち、正解はそれを**取り逃す**形（結果文に「（…の信頼+は取り逃す）」）。
3. **warn 選択肢**は「即時の負効果」か「将来の手戻り(setsFlag)」のどちらかの**下振れ**を必ず持つ。
4. **★0ルール**: どれか1ゲージが0で即バッドエンド（あるある失敗エピローグ）。

## 4. 構造（スクラムのリズム）

- キャンペーン = **Sprint 1 → 2 → 3**。各スプリント = `プランニング → デイリー×5 → レビュー → レトロ`。
- **デイリーだけルーレット**でセグメント抽選（現場/顧客/チーム/トラブル/チャンス）→プールから1問。
  **プランニング/レビュー/レトロは単発**（配列順で最初の出せるイベントを必ず出す＝決定論的）。
- **選択後ミニゲーム**: 開発（タイミング型／手順並べ替えパズル）か、ヒアリング（深掘り質問を選ぶ）を
  セグメントで出し分け。出来(great/good/poor)が**選んだ判断の主正メーターを +1/±0/−1** する（倍率）。
  負効果は不変＝0ルール/トレードオフを壊さない。スキップ＝good。

### 4.1 デイリーの流れ：リモート朝会 → 現地マップ移動（孤独な潜入の演出）

主人公は**たった一人でカルゴ物流に常駐**している。デイリーは次の3ステップ（状態 `travel` を経由）:

1. **ルーレット**を回し、その日の出来事（セグメント）＝今日のイベントが決まる（従来どおり）。
2. **リモート・デイリースクラム**（ビデオ会議の演出）。画面の向こうは本社ルーメンのチーム。
   **役割ごとに観点の違うヒント**が「今日はどの場所が要注目か」を指す（＝行き先誘導／スクラムの役割を自然に学ぶ）:
   - **PO（鷹野）** … 価値・優先度の観点
   - **スクラムマスター（久遠）** … プロセス・障害の観点
   - **開発メンバー（瀬川）** … 技術・事実の観点（※瀬川＝本社で支える新キャスト）
3. **現地マップ**で行き先を選ぶ。場所は5つ：📦倉庫(genba)／🖥️電算室(trouble)／🏢情シス会議室(kokyaku)／
   🗄️総務部（人事・庶務・契約／社内政治）／💻ルーメン開発室【リモート接続】(team)。
   chanceは既定で倉庫、総務部はセグメント既定が無く `location` 明示のイベント（人事・政治系）専用、
   山場イベントは `location` で明示。
   ヒントが指す場所＝今日のイベントの場所へ着くと話が始まる。**外しても「今日は静か」の小景だけでペナルティ無し**
   （ヒント読みを促す）。

- **単発セレモニー（プランニング/レビュー/レトロ＝会議）はマップ無し**。従来どおり「進める」で直接イベントへ。
- 場所⇄セグメントの既定写像と役割ヒントは `src/data/locations.ts`（`LOCATION_BY_SEGMENT` / `hintsFor`）。
  山場（ghost-stock=倉庫・circular=電算室・AI handoff/regression=開発室）と人事・政治系（costcut/faction=総務部）は
  `location`＋手書き `hints` で印象づけ。
- エンジンは `status:'travel'` と `arriveCore`（正しい場所→event／外し→peekLocation のみ）。**travel は一時状態で
  非永続**（リロードすると再びルーレットから。0ルール失敗時はそもそも travel に入らない）。

| Sprint | タイトル | ゴール |
|---|---|---|
| 1 | 現場を知る | 沈黙の理由を突き止める |
| 2 | 仮説を形にする | 誤出荷を減らす最小版を出す |
| 3 | 文化を残す | 成果を本番と人に根付かせる |

## 5. 主軸: ノウハウを隠す現場の自動化（信頼 ↔ トップダウンで分岐）

現場（田淵・山田・橋本）は、自動化＝自分の価値や仕事が奪われる恐れから、勘・暗黙知・**例外処理**を
わざと言葉にしない／隠す。FDEはそれを自動化する任務。**進め方で展開が変わる（毎回必ず分岐）**。

**分岐の仕組み**: 背骨（単発セレモニー）に `requiresFlag` バリアントを既定より前に置き、`proceedCore` が
フラグ一致のバリアントを優先表示＝決定論的。フラグは `s2-retro` で排他的にどちらか一方が立つ。

```
[Sprint1] s1-daily-hideknowhow  「見て覚えるもん」と隠す入口（主軸の提示）
        ↓
[Sprint2 レトロ] s2-retro  ★分岐点（毎回必ず選ぶ）
   ├─ a トップダウンで乗り切る  → flag: topDown   （trust+1 経営の覚え/速い・culture-1, warn）
   └─ b 信頼を築く・一緒に標準化 → flag: genbaTrust（insight+1, culture+1, 機会コストで trust+1 取り逃す）
        ↓
[Sprint3 レビュー]  経路で別イベント
   ├─ topDown   → s3-review-topdown  隠れた例外が漏れ本番デモが崩れる大障害（取り繕う=悪化 / 頭を下げる=挽回の入口）
   └─ genbaTrust → s3-review-trust    現場の勘が組み込まれ例外まで回る（手柄独占=warn / 現場の功績として語る）
        ↓
[Sprint3 レトロ]  経路で結末
   ├─ topDown   → s3-retro-topdown  挽回: 非を認め建て直す(culture+1 部分挽回) / 現場のせいにする(culture-2, 属人化エンドへ)
   └─ genbaTrust → s3-retro-trust    信頼の築き方をチームの財産に＝文化が根を張る
```

- **トップダウンの結末＝手痛い失敗→挽回の余地**（0ルール維持。未挽回は低culture／fail-cultureに寄る）。

## 6.5 不正暴露アーク（自動化×壁壊し → 粉飾露見・隠しフィナーレ）

**FDEの“本物の仕事”が、グループの架空循環取引（§1の粉飾）を炙り出す**。深掘り・連携の選択でのみ手がかりが
立ち、Sprint3 完走後に「暴露の決断」が**フィナーレ層**（ルーレットの外・完走後）として出る＝手がかりを掴んだ
周回でのみ・確実に発火。手がかり無しの周回は従来のメーター駆動エンディング（§8）。

```
[Sprint2] s2-daily-ghost-stock  IT化で実数化→“導入済み”のはずの機材が実在しない（幽霊設備）
            b 出所を追う → flag fraudClue
[Sprint3] s3-daily-circular（要 fraudClue）  壁を壊し取引データを繋ぐ→同じ機材が書類上だけ巡る循環取引
            b 証拠を固める → flag fraudCase（“動かぬ証拠”。暴く結末を左右する）
        ↓ Sprint3 完走（0ルール失敗時はフィナーレ無し＝通常バッドエンド）
[フィナーレ] fraudClue があれば「暴露の決断」（3択は常に提示）
   a 暴く（内部告発） → flag exposed   → fraudCase あり=ED「告発したFDE」（成立・ほろ苦い）／
                                        fraudCase 無し=ED「届かなかった告発」（証拠不足で握り潰される）
   b 黙認（共犯）     → flag complicit → ED「見て見ぬふりのFDE」（成果が粉飾の化粧に使われる）
   c 取り込まれる     → flag coopted   → ED「取り込まれたFDE」（口止めに応じ保身と引き換えに腐る）
```
※ fraudCase は s3-daily-circular で「証拠を固める」を選んだ周回のみ立つ＝**circular の選択が暴露の成否を分ける**。

- 実装: `progression.ts` の純粋ヘルパ `finalEndingFor(meters, flags)`（advanceCore/restoreCore 共用）で
  完走時のED/finalePending を決定。決断は exposed/complicit/coopted を**永続化**し、リロードでも結末が保たれる。

## 8. エンディング（メーター駆動・単一最適解なし）

**通常クリア（Sprint3終了時、上から最初に一致）**:
| id | 条件 | タイトル |
|---|---|---|
| disliked | trust ≤ 2 | 現場に嫌われたFDE |
| orderTaker | insight ≤ 3 | 言われた通り作る人 |
| hero | culture ≤ 2 かつ trust ≥ 4 | ヒーロー止まり（属人化・文化を残せず） |
| trueFde | trust ≥ 7 かつ insight ≥ 6 かつ culture ≥ 6 | 真のFDE |
| decent | 上記以外 | 及第点のFDE |

**0ルール失敗（道中いずれかが0で即）**:
- trust0 → 「もう、来なくていいです」（出入り禁止）
- insight0 → 「結局あれ、何だったんですかね」（使われないシステム）
- culture0 → 「あの人がいないと、何も分からない」（属人化して文化が枯れる）

**フィナーレ専用エンディング（§6.5・fraudClue を掴んで完走した時、通常クリアに代えて）**:
- `finale-expose` 「告発したFDE」（暴く＋fraudCase）／ `finale-expose-weak` 「届かなかった告発」（暴く＋証拠不足）
- `finale-complicit` 「見て見ぬふりのFDE」 ／ `finale-coopted` 「取り込まれたFDE」

## 9. レビュー（矛盾チェック）の観点

物語/設定の一貫性は次を確認（多くは自動テストでも担保）:
- **キャラ呼称の一貫**（情シス→結城／ベテラン→田淵 等。表記ゆれが無いか）。
- **設定の地続き**（StockPilot＝主人公の製品＝使われない不用品、という線が崩れていないか）。
- **メーター原則**（§3）への適合：信頼−は顧客摩擦時のみで理由つきか／warn に下振れがあるか／
  両選択肢が trust+1 等の“支配”になっていないか（機会コスト型か）。
- **フラグ配線**（§5,6）：立てる→回収の前後関係、背骨バリアントの配列順、排他性。
- **エンディング整合**（§8）：閾値と物語のトーンが噛み合うか。
- **用語**：本文の `{{用語}}` が glossary に存在するか（テストで強制）。
- **心得**：各イベントに心得タグ、100箇条カバレッジ維持（テストで強制）。

最新の全イベントの実体は [`docs/EVENT_INDEX.md`](./EVENT_INDEX.md)（再生成可）で突き合わせる。
