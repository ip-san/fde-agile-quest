# 設定の原型カタログ（モチーフ・メモ）

物語の企業・人物・装置が着想を得た**原型（モチーフ）を抽象化**してまとめる。目的は2つ:
1. 既存設定が「何を象徴しているか」を言語化し、ブレずに増築・調整できるようにする。
2. 新しいキャラ・企業・葛藤を作るときの**型（テンプレ）**として再利用する。

> **方針（重要）**: モチーフは**抽象化された原型**として扱う。実在の特定個人をモデルにして
> 断定的・揶揄的に描かない。参照するのは「フィクション作品の類型」「業界のトレンド/通説」
> 「思想・フレームワーク」であって、特定の誰かではない。架空＝架空として書く。

関連: [`STORY.md`](./STORY.md)（設定の正本） / `src/data/chapters/chapter-01/cast.ts`（実装）

## 1. 着想の三本柱（どこから抽象化しているか）

| 柱 | 中身 | 物語での役割 |
|---|---|---|
| **A. FDE / DX のトレンド** | Forward Deployed Engineer の考え方（客先常駐・成果共創）。「SaaS is dead」「導入だけで終わらせない・内製/共創へ」というDX潮流（IPA DX動向 等）。FDE心得100箇条のマインドセット。 | 主人公の使命・ルーメンの転換・学習の主題 |
| **B. 物語の型（フィクション）** | 『The Phoenix Project』型（崩壊寸前の中心事件／ボトルネック／メンター／技術的負債が後で爆発）。池井戸潤型（正論を通すと政治的に損をする社内政治・倍返しのカタルシス）。 | 連続ストーリーの骨格・キャラ類型・部門間対立 |
| **C. 物流ドメイン** | 中堅物流・倉庫現場のリアル（属人化した勘、誤出荷、レガシー、3PL、棚卸差異）。JAVADA ロジスティクス検定の領域。 | 舞台のディテール・現場の本物感 |

## 2. 企業のモチーフ（架空 → 原型 → 抽象核）

| 架空企業 | 着想の原型（抽象） | 何を象徴するか（抽象核） |
|---|---|---|
| **ルーメン**（SaaSベンダー→FDE転換） | “汎用SaaSの成長鈍化に直面し、客先常駐・価値共創モデルへ舵を切るプロダクト企業”という**業界類型**。FDEという働き方の発祥文脈（客先深耕型エンジニアリング）を抽象的に借りる。 | 「ツールを売る」から「成果を共創する」への転換。プロダクト×受託の統合。主人公の出自＝“現場を知らない作り手”。 |
| **StockPilot**（在庫最適化SaaS） | バーティカルSaaS（特定業務特化のクラウド製品）の一般類型。 | “導入したのに使われない”不用品。主人公自身が作ったものが、現場で沈黙している皮肉。 |
| **カルゴ物流**（中堅物流・倉庫。**元は独立企業→フィジカルAI実証用に被買収**） | DX途上の伝統的中堅企業の**類型**（レガシー・紙の承認・属人化・社内政治）＋「外部からグループに買収された“外様”」。**“フィジカルAIの実証の場”として買われたのに、IT化すらされていない**という夢と現実の落差。ルーメンと契約しFDEが入る先。 | 変革の抵抗と本音が渦巻く“現場”。グループから見れば数字を作る駒／実証の体裁。ノウハウを隠す動機に「買収者に明け渡したくない」が重なる。 |
| **ジェネリックロジスティクス**（カルゴ物流の親・物流グループ） | 大手グループの中間持株/事業会社の類型。グループ方針を末端に下ろす中間圧力。 | 上からの自動化要求・粉飾要求の経路。現場 vs 本社の“本社”側。 |
| **ジェネリック電機**（総本山・総合電機・産業機器の大手） | **コモディティ化に追われる旧来型ハードの巨人**が、**フィジカルAI（身体性を持つAI＝ロボット×AI）**へ転換を試みて苦戦し、**政府系（ソブリン/官民）イノベーションファンド**に支えられる業界類型＋**買収を使った架空利益計上（粉飾）の主体**（池井戸型の企業不正の“型”）。 | 失敗が許されないグループ規模のプレッシャーの源泉。「本物の成果より見栄えの数字」が欲しい上層＝既存テーマ「数字を盛る vs 正直」の出どころ。倉庫を“フィジカルAIのショーケース”にしたい動機＝トップダウン経路の背景。将来章の伏線。 |

> 「**ジェネリック＝汎用/コモディティ**」という命名は、テーマ（汎用SaaSも汎用ロボットも価値が薄れる）と
> 響かせた意図的なもの。**架空利益計上（粉飾）・政府系ファンド**は「上からの成果プレッシャー／数字至上」の
> 抽象装置として扱い、実在の特定企業・ファンド・粉飾事件を断定・モデル化しない。

## 3. 人物のモチーフ（架空 → 原型 → 機能）

各人物は「フィクションの類型 × 担うメーター/葛藤」の合成。**特定実在人物ではない**。

| 架空人物 | フィクション類型 | 担う機能（葛藤・メーター） | 抽象核（一言で） |
|---|---|---|---|
| **あなた（主人公）** | Bill 型（火中に投げ込まれる当事者） | プレイヤーの判断主体。客先未経験。 | 作り手が、初めて“現場”に立つ |
| **久遠** | Erik 型（謎めいた師） | FDE心得を“問い”で授けるメンター（メーター外の導き手）。 | 答えは資料の外にある、を体現する声 |
| **橋本** | Brent 型（頼れるが詰まる一点） | 巻き込み(culture)の鍵＝属人化の象徴。制約理論のボトルネック。 | 一人に集中した知＝速さと脆さ |
| **田淵** | 職人気質のベテラン（下町ロケット的現場） | 現場理解(insight)の鍵。暗黙知の持ち主。 | 言葉にしない“勘”＝隠れた要件 |
| **結城** | 板挟みの中間管理職（半沢的な発注側） | 顧客の信頼(trust)の窓口。経営への約束を背負う。 | 引けない約束と現実のあいだ |
| **郷田** | スポンサー経営層（Steve 型） | 投資の当人。成果を急ぐ圧力。 | 数字を見せろ、というプレッシャー |
| **赤城** | 対立する事業部長（Sarah 型／池井戸の敵役類型） | 社内政治の対立軸（コスト削減・3PL・派閥）。 | 正論を高くつかせる“政治の影” |
| **鷹野** | 改革者の経営者 | 転換を決断し主人公を指名（社内側の起点）。 | 旧モデルを捨てる覚悟 |
| **山田** | 現場の名工（流通加工の勘） | 暗黙知のもう一人の担い手。 | 標準化を拒む“見て覚えるもん” |

## 4. 物語装置のモチーフ（テーマ → 元になった考え方）

| 装置（物語の仕掛け） | 元の考え方（抽象） |
|---|---|
| **沈黙する基幹システム**（使われない不用品） | 「導入＝成果ではない」「使われたかを見ろ」。アダプション/習慣化の問題。 |
| **ノウハウを隠す現場の自動化**（主軸の分岐） | 自動化が雇用・熟練の価値を脅かす恐れ → 信頼で開くか、力で奪うか。経験主義 vs トップダウン。 |
| **橋本＝ボトルネック** | 制約理論（全体の速さは最も詰まった一点で決まる）／属人化の解消。 |
| **誤ったKPI→手戻り**（wrongKpi） | 「上が崩れたら下は全部やり直し」。仮説のミルフィーユ／Whyの最上段。 |
| **AI過信→モデル退化**（aiOverreliance） | 「評価なき生成はただの祈り」「AIは速度をくれるが責任はくれない」。検証/ガバナンス。 |
| **部門間対立**（責任なすりつけ・手柄横取り・派閥） | 池井戸型の社内政治。正義と政治のトレードオフ。 |
| **大障害→挽回** | DevOps的フィードバックと学習。「炎上は授業料つきの実地訓練」。 |
| **架空循環取引→暴露**（不正暴露アーク） | 会計士が不正を暴く小説の構図＋**実在の循環取引型粉飾の“型”**（同じ商材を書類上だけ巡らせ売上を水増し／幽霊在庫・幽霊設備）を、**産業用ロボット・フィジカルAI機材×物流**に翻案。FDEの可視化・組織連携が不正を炙り出す。**特定企業・事件はモデル化しない**（型のみ着想）。 |

## 5. 新しい設定を作るための“原型カード”テンプレ

キャラ・企業・葛藤を増やすときは、この型を埋める（抽象核を先に決める）。

**キャラ原型カード**
```
- 名前: （和風で平易に）
- フィクション類型: （Bill/Brent/Erik/Steve/Sarah/職人/中間管理職… の合成）
- 所属・立場: （顧客/現場/社内/メンター/敵役）
- 担うメーター: trust / insight / culture / なし
- 中心の葛藤: （何と何の板挟みか）
- 抽象核（一言）: （その人物が象徴するもの）
- 呼称ルール: （表記ゆれを防ぐ正式名）
```

**企業/舞台原型カード**
```
- 名前:
- 業界類型: （DX途上の中堅 / バーティカルSaaS / 受託 …）
- 象徴する課題: （レガシー / 属人化 / 社内政治 / 成長鈍化 …）
- FDEが勝ち取る相手・指標: （誰の信頼 / どのKPI）
```

**葛藤（イベント）原型カード**
```
- 状況: （現場/顧客/チーム/トラブル/チャンス のどれか）
- 速く見せる選択（warn）: trust+1 の機会コスト型 / 即時の負 or setsFlag
- 筋を通す選択: insight・culture+（必要なら顧客摩擦で trust−、理由を結果文に）
- 体現する心得: （precepts.ts の番号）
- 必要なら分岐フラグ: setsFlag / requiresFlag
```

> 守るべき不変条件は [`STORY.md`](./STORY.md) §3（メーター原則）と §9（矛盾チェック）。
> 原型カードを埋めたら、イベントは EVENT_INDEX の体裁に落とし込む。
