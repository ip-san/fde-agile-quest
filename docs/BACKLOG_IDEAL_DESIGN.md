# バックログ層 理想設計ドキュメント（専門家パネル考証済み）

**目的**：バックログ層を物語の背骨に統合し、「触っても話が変わらない側パネル」を解消する。
**前提**：診断レポート（`docs/BACKLOG_DIAGNOSIS.md`）。エンディングは3メーターだけで決まり、バックログ層は結末に未接続＝これが"機能してない感"の主因。
**考証**：agile-expert / fde-expert / learning-designer の3パネルでレビュー済み（本書はその合議結果）。
**本書の位置づけ**：設計の合意形成用。実装設計・物語落とし込みは次段（narrative-designer / 実装）。

---

## 0. 設計思想（全員一致で正しい背骨）

2層（ルーレット＝物語／カンバン＝バックログ）を**1つの経験ループの2つの顔**にする。プロダクトゴール「**小さく作り、大きく学び、太く残す**」を3機構に対応させ、バックログを章の通し糸にする。

```
   現場イベント（物語）
        │ ① 発見 = 真のPBIがバックログに現れる（大きく学び）
        ▼
   プロダクトバックログ ──② 優先順位＝価値判断（小さく作り）──▶ POが裁定／物語が遅れて押し返す
        │                                                          ▲
        │ forecast → 着手(AI) → レビュー → Done                    │
        ▼                                                          │
   Done / 負債 / カバレッジ ──③ Shipが物語ビートを生む（実装主義）─┘
        │
        ▼
   ④ 去り際の"残した仕組みの実体"がエンディングを書く（太く残す）
```

**この背骨は3パネルとも🟢**（agile G2/G3、fde 🟢8、learning 用語の体験紐付けが上がる）。ただし**4機構すべてに誤教育/非リアル/単一正解化のリスク**があり、以下のガードレール込みで初めて「理想」になる。

## 1. 機構①：発見 → バックログ（大きく学び）

**意図**：初期バックログ＝顧客の言い値（在庫画面に予測機能＝間違ったKPI）。真に効くPBIは現場ヒアリングで掘って初めて出現。「答えは資料の外にある」を機構化。

**🟢 妥当（全員）**：顧客の言い値が間違ったKPIで真の課題は別、という構図はFDE案件のリアルそのもの（fde 🟢7）。`wrongKpi`/`orderTaker`エンディングと一貫（agile G3）。

**ガードレール（考証で追加）**：
- **🔴 一発で真実は湧かない（fde 🔴2）**：現状の `discoversPbi→即追加` は成功一直線で「現場に入れば真実が湧く打ち出の小槌」に見える。**信頼ゲート**（trust が一定貯まって初めて深いPBIが出る）を1段挟む。空振り・偽の手がかりは"入れすぎると教育価値を損なう"ので第1章では最小限。
- **🔴 見逃しは沈黙させない（learning 🟡-1）**：掘り損ねても今は何も起きず、学習者は失敗に気づけない。**二段化**＝「掘れば早く安く出る／掘らなければ後段で高コストの強制イベントとして顕在化」。例：聞き逃した夜勤引き継ぎ漏れ（`pbi-disc-night-shift`）が Sprint3 で誤出荷トラブルとして強制発火し「あの時聞けていれば」と振り返らせる。→ 詰ませず、機会コストを体感させる。
- **🟡 リファインメントの明示（agile Y1）**：発見直後のPBIは estimate が不確か。Ready 化（分解・見積り）を一段挟むと "emergent な PB" が正確になり、フォーキャストの不確実性（機構②）と噛む。

**触る場所**：`backlog.ts:58-65 revealPbi`、`DISCOVERABLE_BACKLOG`（Sprint2/3 に発見源を追加）、events の `discoversPbi`。

## 2. 機構②：優先順位 → 物語が押し返す（小さく作り）

**意図**：レビュー容量がボトルネック＝全部入らない。情シス（予測機能）と現場（誤出荷修正）が引っ張り合い、下げた方が反応イベントを生む。今のPO自動補正（正解順に直す）をやめ"葛藤を残す"。

**🔴 最重要ガードレール — ここを誤ると学びが死ぬ（3パネル全員が警告）**：

- **PO の確定構造は絶対に残す（agile R3）**。消すのは「PO が一意の正解順を知っている描写」であって「PO が決める」構造ではない。スクラムガイド：優先順位の accountability は PO に集中。**プレイヤー＝FDE は propose（提案・論拠提示）に留め、decide は PO キャラ**。`floated` の仕組みは残し、戻す理由を「sprintHint一致」という機械ルールから「今スプリントの **Sprint Goal** に資するから」という**価値の言葉**へ変える。
- **機会コスト型・両側に代償（learning 🔴-1）**。「情シスを下げる＝常に良い」になると、`wrongKpi`と同型の"⚠＝地雷"を即学習し、2周目以降は思考停止で現場側を上げる**単一最適手**に収束＝トレードオフ学習が死ぬ。**どちらを下げても別メーターで代償**を出す（chapter-01.ts:30-32 の「トレードオフ＝機会コスト型」原則をバックログ並べ替えにも適用）。
- **政治コストは非対称（fde 🔴1）**。発注者＝情シス（結城係長）を下げる痛みは、現場ベテランを下げる痛みより桁違いに重い（経営に約束済みで引けない発注者だから）。対称な「引っ張り合い」で均すとFDE最大のジレンマ（正論＝現場優先が契約者の面子を潰す）が薄まる。→ 下げた対象で反応の質を変える：情シス↓＝trust− ＋面子イベント、現場↓＝insight− ＋後の手戻り。
- **反応は遅れて響く（fde 🟡5）**。「次のデイリーで即発火」はご都合に見える。既存の機会損失アーク（STORY §4.1 missedHearing→次スプリントで回収）の時間差に揃える。

**触る場所**：`backlog.ts:111-145 reviewBacklogProposal`（補正ルールを価値の言葉へ）、Sprint Goal の昇格（§5）、反応イベント（events）。

## 3. 機構③：Ship → 物語を生む（実装主義）

**意図**：Done が物語ビートを発火。misship-MVP出荷→現場の反応・誤出荷が下がる。浅い(quick)レビューで負債を溜めて出荷→後日 `demofail`/`debt` トラブル回。

**🟢 妥当**：AIが着手を担い人がレビューに専念するコスト構造（GEN_TOKEN_COST=80 vs REVIEW_CAPACITY=6）は2026年のAI開発の経済としてリアル（fde 🟢9）。quick/thorough の機会コストは既によく出来ている（learning 🟡-3）。

**ガードレール**：
- **部分Doneで物語を発火させない（learning 🟢/完了の重み）**：`prog >= estimate` の全Doneでのみビート発火。9割で話が進むと「完了の重み」の学びが空洞化。
- **負債は一律ペナルティにしない（learning 🟡-3）**：glossary は技術的負債を「意図的に借りて素早く返すのが上手な使い方」と定義。「浅くても期日に間に合わせ顧客信頼を守り、後で計画的に返す」が**正解になりうる局面を1つは残す**。一律地雷化は単一正解化（🔴-1と同型）。
- **AI過信の罠の置き場所（fde 🟡4）**：本当の怖さは「浅いレビュー」より「もっともらしく動くが要件を微妙に外したAI出力を信じて素通し」。深さ(quick/thorough)の量だけでなく「**鵜呑みにしたか**」の質の軸を `aiOverreliance` と連動させる。AIが大量生産→人のレビューが追いつかず素通し、という負の循環を罠に。
- **DoD と Done の分離（agile Y3）**：現状 "DoD達成＝Done" は実体が「見積り消化」でDoDではない。**quick で Done にした項目は DoD 未達フラグ付きで出荷**され後日トラブルを呼ぶ、と明示すると「Done の定義を妥協すると負債になる」という核教訓が立つ。

**触る場所**：`backlog.ts:195-219 reviewItem`、demofail/debt トラブルイベント（既存・直近コミットで種別明示済み）、`aiOverreliance` フラグ。

## 4. 機構④：実体 → エンディング（太く残す）

**意図**：結末を3メーターでなく「去った後に残るシステムの状態」（文化系PBI handoff/onboarding/monitoring を Ship したか、負債/カバレッジ）で読む。

**🔴 最重要ガードレール — "Done数"は使わない（3パネル全員）**：
- **出力カウントで「太く残す」を近似しない（agile R1）**。`delivery=Done数/6` を結末直結させると「6個出せば勝ち」＝velocity至上主義を教える。スクラム：velocityは目標でなく予測データ。**「Done数」を判定から外し、「どの成果を残したか（質）」に一本化**。
- **3メーター主軸＋上位エンディングのAND関門型（learning 🔴-2）**。バックログ成果を並列加点すると「盤面を回す作業ゲー」に最適化し、「正しさだけでは足りない」という最重要メッセージ（hero/orderTaker）が薄まる。**3メーター判定は主軸のまま残し、`trueFde` のような上位結末にのみ「文化系PBIをShip＆定着」をAND条件で追加**。スクラム実践を「FDE心得だけでは届かない最後のひと押し」として直列に置く。
- **二重カウント回避（learning 🔴-2）**：文化PBI Ship→culture+→結末、かつ Ship実体→結末、だと片方が形骸化。**メーター＝道中の関係性／実体ゲート＝去り際のレガシー**と役割分離する。
- **Ship≠定着（fde 🟡6）**：手順書をShipしても誰も読まない・更新しないのが物流現場のあるある（fail-culture）。「文化PBIをShip」を結末直結させると「成果物を作れば文化が残る」錯覚を肯定してしまう。**Ship は必要条件、"現場が実際に使い始めた／属人化（橋本）が解けた"という使われた証拠とセット**にする。
- **可視化＝即暴露の短絡を避ける（fde 🔴3）**：fraudClue を「ダッシュボードShip」一段に紐付けない。ダッシュボードShip が立てるのは fraudClue（違和感の入口）まで。fraudCase（動かぬ証拠）は既存の突合せ系（ghost-stock＋keiri連結突合せ、STORY §6.5）に残す。「作ったから**気づける土俵に立てた**」≠「作ったから**暴ける**」。

**触る場所**：`chapter-01.ts:124-159 ENDINGS.match`（AND関門追加）、FAILURE_EPILOGUES（fail-culture を実体連動へ）、fraudClue 配線（一段化しない）。

## 5. 横断：昇格すべきスクラム概念（agile 指摘）

データ上は存在するのに機構に昇格していない4概念。これらを立てると教育精度が上がる。

| 概念 | 現状 | 昇格案 |
|---|---|---|
| **Sprint Goal** | `sprintHint`（項目タグ）に格下げ。データ上は backlog.ts コメントに存在 | 単一の目的文として一級市民化。フォーキャスト編集を「Goal に資するか」で見せる。②の綱引きの山場＝「どの項目を削っても Goal は守る」判断 |
| **DoD** | `prog>=estimate`（バーンダウン完了）に潰れている | 品質の門として分離。quick-Done＝DoD未達＝undone work→後日トラブル（③と連動） |
| **Refinement** | 機構①に内在するが無名 | 発見PBIを Ready 化（見積り確定）する一段（①と連動） |
| **Retro** | `cultureDelta`（systemが勝手に採点）が代用、チームの主体性なし | レトロで決めた改善（DoD上げ/WIP下げ）が次スプリントの容量・負債蓄積に効く。culture メーターを**プロセス改善として**盤面に繋ぐ |

## 6. オンボーディング：1スプリント1機構の漸進開放（learning 🟡-2）

4機構を Sprint1 から全開にすると、管理変数（並べ替え＋WIP＋レビュー深さ＋発見＋負債＋3メーター＋顧客KPI）が一気に増え、用語も高密度（glossary 60語超）で初学者が溺れる。

→ SPRINTS（現状3スプリント均質）の**機構開放カーブを意図的に非均質に**する：
- **Sprint1**：「発見→バックログ（①）」だけを密結合。片肺で慣れさせる。
- **Sprint2**：優先順位の葛藤（②）＋Ship物語化（③）を開放。
- **Sprint3**：実体結末（④）＋レガシーの賭けを開放。

## 7. リスク／非ゴール

- **非ゴール**：バックログの完全必須化（物語目当ての導線を硬くしうる）。あくまで「非ブロッキング＋結果フィードバック」哲学（learning 🟢）は維持。
- **リスク**：盤面状態で分岐するイベント variant の組み合わせ爆発。→ 分岐は「下げた対象（情シス/現場）」「DoD未達フラグ」「文化PBI Ship有無」など**少数の状態フラグ**に集約し、本文 variant を最小化。
- **リスク**：セーブ互換（既存 persisted）。新フラグは後方互換デフォルトで吸収。

## 9. 総監督への確認事項（合議で割れた/判断が要る点）

1. **機構②の葛藤の強度**：fde は「政治コスト非対称（情シス↓が重い）」を求め、learning は「両側に代償（対称な機会コスト）」を求める。**非対称だが両側に代償**（情シス↓＝trust大／現場↓＝insight＋遅延手戻り）で両立できるが、第1章の難度として葛藤をどこまで効かせるか。
2. **機構①の空振り/信頼ゲート**：fde はリアリティのため空振りを求め、learning は教育負荷のため最小化を求める。第1章は「信頼ゲート1段＋空振りなし」で始めるか。
3. **レトロの昇格**：最も薄い領域。今回の再設計に含めるか、別タスクに切るか。
4. **着手範囲**：背骨（①④）だけ先に実装設計へ落とすか、本書全体を物語設計（STORY/cast 更新）まで一気に固めるか。
