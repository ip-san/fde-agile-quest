# story-review — 物語・設定の矛盾チェック

このプロジェクトのレビューは**コードより物語・コンテンツの整合性**が主。観点は
`docs/STORY.md` §9（矛盾チェック）／§3（メーター原則）に言語化されている。

**役割分担（最重要）**: 自動テスト（`src/engine/game.test.ts` ほか）が**構造**は守る。
この skill は**テストが見ない「文章の中身・意味・トーン」**だけを担当する。重複して
構造チェックをやり直さない。

## 真実源（この順で参照）

1. `docs/STORY.md` — 設定の正本（人が編集する版）。§3 メーター原則 / §9 矛盾観点
2. `src/data/chapters/chapter-01/cast.ts` — 物語バイブル（中心事件・3幕・キャラ）
3. `docs/MOTIFS.md` — 各設定の抽象核（ブレ判定の基準）
4. 実装の正本: `src/data/chapters/chapter-01/`（events-sprint*.ts / cast.ts / names.ts）

## 手順

### 1. 機械チェックを先に通す（ここは skill の仕事ではない＝テストに任せる）

```bash
npm run typecheck
npm test
npm run gen:index   # EVENT_INDEX.md を再生成
git status --short docs/EVENT_INDEX.md   # 差分が出たら index が stale だった = コミット漏れ
```

- テストが落ちたら、それは**構造の問題**。内容レビューの前に直す/報告する。
- `gen:index` 後に `docs/EVENT_INDEX.md` が変化したら、索引が現状とズレていた合図。再コミットを促す。

### 2. レビュー範囲を決める（差分スコープ）

```bash
git diff main...HEAD --stat        # ブランチで触れたファイル
git diff main...HEAD -- src/data/chapters/   # イベント本文の差分
```

原則、**変更のあったイベント・設定だけ**を深く読む。全120イベントは読まない
（全体傾向の確認が要るときだけ `docs/EVENT_INDEX.md` を俯瞰）。

### 3. 文章の中身をレビュー（テストで拾えない観点だけ）

各差分イベントの `narrative` / 各 `Choice.label` / `resultText` を読み、次を確認:

**A. メーター原則の“意味”（STORY.md §3）— 構造ではなく文章で判定**
- [ ] **信頼−には理由が結果文(`resultText`)に書いてあるか**（§3-1）。顧客摩擦
      （要望後回し/納期割れ/粗いものを見せる/耳の痛い事実/押し返し/面子を潰す）が
      文章で説明されているか。理由なき trust− は原則違反。
- [ ] チーム/プロセス系の“良い選択”が信頼を下げていないか（§3-2）。
- [ ] **トレードオフが機会コスト型か**（§3-2）。片方が支配的（両メーターで勝ち）に
      なっていないか。「速く見せる」側が trust+1（進捗が見える）を持ち、筋を通す側は
      それを**取り逃す**形か。結果文に「（…の信頼+は取り逃す）」の含意があるか。
- [ ] **warn の下振れが意味的に妥当か**。テストは「即時負 or setsFlag が存在する」
      ことだけ見る。skill は「その負/手戻りが物語的に筋が通るか」を見る。

**B. キャラ・設定の地続き（STORY.md §9 / cast.ts / MOTIFS.md）**
- [ ] **呼称の一貫**: 情シス→**結城**／倉庫ベテラン→**田淵**／総務→**守屋**／
      人事→**新田**／経理→**間宮**。地の文で部署名と人名が混在していないか。
      口調・一人称・敬称のドリフトが無いか（names.ts の機構はあるが、執筆時の
      取り違えは検知しない）。
- [ ] **設定の地続き**: 「StockPilot＝主人公が作った＝現場で使われない不用品」の線が
      崩れていないか。粉飾/フィジカルAI/政府系ファンドを本文に出す場合、平易に補足され
      実在の特定企業・事件を断定していないか（MOTIFS.md §2 の方針）。
- [ ] **抽象核との整合**: 新規/改変キャラが MOTIFS.md §3 の型から逸脱していないか。

**C. フラグ配線の物語的整合（STORY.md §5,§6,§6.5）**
- [ ] 立てる(`setsFlag`)→回収(`requiresFlag`)が**配列で前後**になっているか
      （テストは存在を見るが、新規イベントの配置順は要目視）。
- [ ] 分岐（topDown/genbaTrust の排他、fraudClue→fraudCase）の物語が破綻していないか。

**D. エンディング整合（STORY.md §8）**
- [ ] 閾値（trust≤2 で disliked 等）と結末文のトーンが噛み合うか。

**E. 用語・トーン**
- [ ] `{{用語}}` が文脈に自然か（過剰な詰め込みでないか）。新語は `glossary.ts` に
      平易な解説があるか（本文で使うならテストが存在は強制する）。
- [ ] 語り口が『The Phoenix Project』／池井戸潤テイスト（正論が政治的に損をする
      ジレンマ）から外れていないか。
- [ ] **比喩は一読で意味が取れるか**（STORY.md §9）。大仰で空回りする／抽象的すぎて
      意味を考え込む比喩（「砂のまま残った」「時間が溶けた」「数字に滲む」「沈黙の輪郭」等）
      は🟡で平易化を促す。像が結ぶ比喩・伏線定型（火種・§6.5「違和感／粉飾の輪郭」）は残す。
- [ ] **メタ自己言及が無いか**。地の文が自分を「物語／ゲーム／キャンペーン」と呼ぶ、
      プレイヤー表示文で主人公を「主人公」と書く等の 4th-wall 破りは🔴で是正を促す。

### 4. 報告

指摘は**ファイル:該当イベントID**で示し、STORY.md の該当節（§3-1 等）を根拠に挙げる。
重大度を分けて簡潔に:
- 🔴 **原則違反**（メーター原則・設定の地続き・呼称矛盾）= 直すべき
- 🟡 **要検討**（トーン・用語の自然さ・機会コストの弱さ）
- 🟢 **良い点**（残すべき意図的な設計）

最後に「テストは通っているか」「EVENT_INDEX は最新か」を1行で添える。

## やらないこと（スコープ外）

- テストが既に強制する構造チェックのやり直し（100箇条カバレッジ・心得タグ必須・
  warn 下振れの存在・フラグ挙動・glossary 存在・メーター clamp）。
- イベントの新規追加・本文の書き換え（レビューであって執筆ではない）。指摘に留め、
  修正は別途依頼を受けてから。
