# director-feedback — 総監督フィードバックの取り込み

総監督の指摘を、**一回限りの修正で終わらせず、該当エージェントが恒久的に覚える観点**に変えて
`.claude/agents/learnings/<agent>.md` に追記する。これにより loop が回るほどチームが総監督の好みに収束する。

## 大原則
- **一般化する**: 目の前の1箇所を直すのではなく、再発を防ぐルールにする。
  - ✕「この朝会のセリフを直して」→ ◯「SMの口上で同じ語尾を3連続で繰り返さない」
- **責任の所在に記録する**: 誰の仕事の観点かを見極めて、その人のノートに書く（複数可）。
- **既存ノートと統合する**: 重複や矛盾があれば版を更新（古い行は「（YYYY-MM-DD 更新）」で残す）。
- **勝手に作品を直さない**: このスキルの仕事は「観点の記録」。実際の修正は次回以降、該当エージェントが
  loop の中で行う（その場で直してほしいと明示されたら別途対応）。

## 手順

1. **不満の本質を聞き取る**: 何が・なぜダメだったか。曖昧なら1〜2問だけ確認（AskUserQuestion）。
   具体例（before）と、できれば望ましい姿（after）を押さえる。

2. **責任エージェントを特定**（複数可）:
   | 不満の対象 | 記録先 |
   |---|---|
   | 言い回し・トーン・テンポ・キャラの声 | `narrative-designer` |
   | 操作感・見た目・モバイル・配色・モーション | `ux-engineer` |
   | ドメインの考証漏れ・事実誤り | 該当専門家（`ai-dx`/`fde`/`agile`/`logistics`/`robotics`） |
   | 設定矛盾・呼称ブレの見逃し | `story-reviewer` |
   | 学びが薄い・トレードオフに正解がある | `learning-designer` |
   | 設計・可読性・型の負債 | `code-reviewer` |
   | ゲートの見落とし・緩め | `quality-gatekeeper` |
   | 一手の選び方・スコープ・優先度 | `showrunner` |

3. **観点に一般化して追記**: 該当 `.claude/agents/learnings/<agent>.md` に、日付付きで1項目を追記する。
   ```
   ## YYYY-MM-DD
   - [観点] 〜しない / 〜する。（背景: 総監督指摘の要旨）（例: before→after の要点）
   ```
   - 日付は環境の「Today's date」を使う。
   - 既に近い観点があれば新規追加せず統合・明確化する。

4. **報告**: 「誰のノートに・どんな観点を記録したか」を簡潔に提示。今後その観点が作業前に必読される旨を添える。

## やってはいけないこと
- 観点を一般化せず「その場限りの修正メモ」を貼ること（再発防止にならない）。
- 責任が曖昧なまま全員に同じ指摘をばら撒くこと（記録先を見極める）。
- ノートの過去の観点を黙って削除すること（版で残す）。
