# loop 完全自走ドライバ（自己給餌・背圧式 / GO 廃止版）

`loop-autonomous.md`（GO ゲート版）の**完全自走バージョン**。人間の GO（`Approved` 承認）を外し、
**playtest が仕事を自分で発見してボードに積み、エンジンが実装して PR まで出す**——夜が回るほど
ゲームが自律的に磨かれる自己給餌ループ。

> **GO を外した代わりの安全装置 = 出口の背圧**。入口で人間が承認する代わりに、
> **未マージ PR が上限に達したら自動で生産を止める**。人間のマージ速度にシステムが同期し、
> 寝ている間に PR が溢れない。これが「完全自走」を暴走でなく自己 throttle にする要。

## 起動方法（総監督）
```
/loop @.claude/agents/loop-autonomous-playtest.md
```
（interval 無し＝dynamic モード。`ScheduleWakeup` で再起動を予約。ローカル方式＝セッション/PC を起動したまま）
> 朝に見るのは **ボードの `In review`（マージ待ち PR の一覧）**。GO 列はもう使わない。
> 止めるのは `/loop` 停止。完全に手綱を握り直したいときは GO 版 `loop-autonomous.md` に戻す。

## 外さないガードレール（GO 以外は全部残す＝これが本当の安全網）
- **作業は feature ブランチ・PR で提案のみ**。**main 直 land / `git push --force` / 自動マージは capability で禁止**。
  **マージは人間**（PR がレビューゲート＝完全自走でも最終ゲートは人間のマージ）。
- **正本絶対視**: `docs/STORY.md` / `cast.ts` 既存設定の改変・retcon・新キャラ・新章・破壊的変更は
  **自走では絶対に実装しない**。発見段で見つけたら Issue リスク欄に明記 → 実装段で `Blocked` へ移し人間へ。
- **品質ゲート不緩和・maker/checker 分離**（チーム憲章）。

## キュー（GO 廃止後の仕事の並び）
- `Approved` ゲートは**使わない**。キュー = **open な `loop` Issue のうち `Blocked`/`In review`/`Done` 以外を
  FIFO（issue 番号昇順）**。取得は `node .claude/agents/loop/next-batch.mjs`（最大 `config.limits.batchSize` 件）。
- `Backlog` 列は「発見済み・未着手」の置き場として残るが、**人間の承認は挟まずそのまま着手対象**。

## 1ティックの動作（毎回これを実行）

### A. 背圧チェック（最初に必ず・GO の代替・暴走の最終防壁）
```
node .claude/agents/loop/backpressure.mjs
```
- `{"blocked":true,...}`（未マージ PR ≥ 上限 M、既定10）→ **新規実装をしない**。エージェントを1体も起動せず、
  `ScheduleWakeup(delaySeconds: 1800, reason: "背圧: マージ待ち PR が上限。人間のマージ待ち")` で予約して**即終了**。
  ＝人間がマージして PR が減るまで生産停止。
- `{"blocked":false, count:N, limit:M,...}` → B へ。**`available = M - N` をメモ（C-2 のバッチ上限調整に使う）**。
- `{"ok":false,...}` → ボード/権限異常。`RUNLOG.md` に1行残し idle 予約して終了（無人なので止めない）。

### B. バッチ取得
```
node .claude/agents/loop/next-batch.mjs
```
- `{"hasTask":true, tasks:[...], queued:N}` → **C（バッチ実装）**へ。`tasks` が作業指示書群。
- `{"hasTask":false}` → キューが空 → **D（発見）**へ。
- `{"hasTask":false,"error":...}` → 異常。`RUNLOG.md` に残し idle 予約して終了。

### C. バッチ実装（並列対応）

B で返ってきた `tasks`（1〜3件）を**ドメイン分類してぶつからない組を並列で実装**する。

#### C-1. ドメイン分類（showrunner がタイトル＋本文の冒頭を読んで判定）

| ドメイン | 判定キーワード（タイトル優先） | 主なファイル |
|---|---|---|
| `s1` | `s1-` / `Sprint1` / `S1` のみ（S2/S3 言及なし） | events-sprint1.ts |
| `s2` | `s2-` / `Sprint2` / `S2中盤` / `S2後半` など（S1/S3 言及なし） | events-sprint2.ts |
| `s3` | `s3-` / `Sprint3` / `S3` のみ（S1/S2 言及なし） | events-sprint3.ts |
| `ux` | `ResultModal` / `コンポーネント` / `UI` / `画面` / `操作感` / `a11y` | src/components/ |
| `shared` | 複数スプリントをまたぐ / 分類不能 | 複数ファイル |

#### C-2. 並列グループ判定
- **並列グループ（異なるドメイン）**: 例 `s2 + s3`、`s3 + ux`、`s1 + s2 + ux` → **並列実装**
- **直列グループ（同一ドメイン or `shared`）**: 先頭1件のみ実装し、残りは次ティックへ（FIFO で来る）
- A の `available` が1の場合はバッチ先頭1件のみ（並列しない）

#### C-3. 実装ディスパッチ

**並列グループ（2件以上）**:
- 各 issue に対して `Agent` ツールを**同一ターンで並列呼び出し**（複数の tool use を1メッセージに）
- 各 Agent には `isolation: 'worktree'` を指定（独立ワークツリーで同時実行・ファイル衝突なし）
- 各 Agent は `loop-runbook.md` 手順1〜5 を**1件分**独立実行（着手 → maker → checker → gate → PR）
- 全 Agent の完了を待ってから RUNLOG.md を更新

**シングル（1件のみ）**:
- 通常通り単一 Agent で処理（worktree・Workflow 不使用・現行と同じ）

**不変条件（並列でも変わらない）**:
- 正本改変/破壊的/受け入れ条件が曖昧 → 着手せず `Blocked` へ移しコメント
- 品質ゲート（`npm run check`）は各ワークツリーで独立実行
- マージは人間（PR 作成まで）

### D. 発見フェーズ（キューが空のとき＝アイドルを発見に変える）
`playtest-triage` skill を**1回実行**する（手順の正本は skill。**ここで再記述しない＝重複を作らない**）。
- 自走から呼ぶときの不変条件（重複除外・上位 N 件＝config `limits.discoverPerTick`・検収で閉じる）は
  skill の「**完全自走ドライバから呼ばれるとき**」節に従う。
- ティック制御（ここがドライバの仕事）:
  - 新規 Issue を積めたら → **A に戻る**（次ティックで実装へ）。
  - 発見しても**新規ゼロ**（全部重複 or 飽き点が出尽くし）→ `ScheduleWakeup`（idle 長め）で予約して終了。

### E. アイドル
背圧で止まった/キューも発見も空 → `ScheduleWakeup(delaySeconds: 1800, ...)` で予約して終了。次ティックは A から。

## 暴走防止の三層（GO が無くても破綻しない理由）
1. **出口の背圧（A）**: 未マージ PR ≥ M で生産停止。人間のマージ速度がそのままスループット上限になる。
2. **発見のレート制限と重複除外（D）**: 1ティック N 件まで・既知は再起票しない・検収で閉じる。
3. **正本/破壊の自動 Blocked（C）**: 自走では正本を触らない。判断が要るものは人間に上げる。
- さらに **マージは人間**（capability deny）＝本番反映の最終ゲートは常に人間。

## 完全自走で増えるリスク（運用者が知っておくこと）
- **事前操舵の喪失**: 人間は実装前に却下できず **PR でしか拒否できない**。緩和＝背圧 M で PR を溢れさせない＋
  PR 本文に「どの局面の何の飽きを・なぜ直したか」を必ず書く＋朝に `RUNLOG.md` の1行ダイジェスト。
- **過剰チューニング**（目新しさ偏重）: 監修に `learning-designer` を必ず残し「面白いが学びゼロ」を相殺（C のエンジンに既在）。
- **コスト**: 並列化でトークン消費が増える。緩和＝発見はキュー枯渇時・N 件まで、背圧で停止。

## 収束（いつ止まるか）
- 上限は設けない。背圧で詰まればアイドル、キューも発見も尽きればアイドル、GO 待ちはもう無い。
- 完全に止めるのは総監督が `/loop` を停止したとき、または GO 版ドライバへ戻したとき。
- ティックごとに `findings.md` / `RUNLOG.md` を更新し、朝に人間が `In review` と合わせて追えるようにする。
